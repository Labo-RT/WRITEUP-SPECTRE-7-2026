           Protocole d'Authentification par Défi-Réponse S7-RAP
                  (Spectre 7 Radio Authentication Protocol)


1.  Introduction

    Ce document décrit le protocole S7-RAP, utilisé pour l'authentification
    des badges d'accès sur les terminaux radio Spectre 7. Ce protocole est
    conçu pour fonctionner sur du matériel embarqué à ressources limitées.

    Il repose sur un mécanisme de défi-réponse (challenge-response) afin
    d'éviter la réutilisation de credentials interceptés.


2.  Terminologie

    CLIENT   : entité demandant l'accès (badge)
    SERVEUR  : entité contrôlant l'accès (lecteur)
    SECRET   : séquence de caractères ASCII connue uniquement du CLIENT
               et du SERVEUR
    CHALLENGE: entier aléatoire généré par le SERVEUR à chaque session
    SESSION  : échange complet entre un CLIENT et un SERVEUR


3.  Niveaux d'accès

    Le protocole distingue deux niveaux d'accès selon l'identifiant
    transmis en phase AUTH :

        USER   : accès standard — le SERVEUR confirme l'ouverture
                 par un message OK
        ADMIN  : accès privilégié — le SERVEUR renvoie en plus
                 un FLAG chiffré


4.  Architecture générale

    Les messages sont des chaînes ASCII terminées par un saut de ligne.
    Le canal radio opère en half-duplex.

    Diagramme de séquence — accès USER :

        CLIENT                                  SERVEUR
          |                                        |
          |--- AUTH:USER ------------------------->|
          |                                        |
          |<-- CHALLENGE:<entier> -----------------|
          |                                        |
          |--- RESP:<hex> ------------------------>|
          |                                        |
          |<-- OK ----------------------------------|
          |                                        |

    Diagramme de séquence — accès ADMIN :

        CLIENT                                  SERVEUR
          |                                        |
          |--- AUTH:ADMIN ------------------------->|
          |                                        |
          |<-- CHALLENGE:<entier> -----------------|
          |                                        |
          |--- RESP:<hex> ------------------------>|
          |                                        |
          |<-- FLAG:<hex> --------------------------|
          |                                        |

    La session est fermée par le SERVEUR immédiatement après la phase 4.


5.  Spécification des messages

5.1 AUTH

    Syntaxe :   AUTH:<identifiant>
    Direction : CLIENT → SERVEUR
    Rôle :      Ouverture de session et identification du CLIENT

    Identifiants reconnus : USER, ADMIN
    Tout autre identifiant provoque un DENIED immédiat.

5.2 CHALLENGE

    Syntaxe :   CHALLENGE:<n>
    Direction : SERVEUR → CLIENT

    <n> est un entier aléatoire dans l'intervalle [0, 255].
    Généré à chaque session. Ne peut pas être réutilisé.

5.3 RESP

    Syntaxe :   RESP:<hex>
    Direction : CLIENT → SERVEUR

    <hex> est la concaténation des octets résultant de l'opération
    décrite en section 6.1, encodés en hexadécimal sur 2 caractères
    chacun, sans séparateur.

    La longueur de <hex> est toujours égale à 2 * len(SECRET) caractères.

5.4 OK

    Syntaxe :   OK
    Direction : SERVEUR → CLIENT
    Condition : identifiant USER + RESP correct

5.5 FLAG

    Syntaxe :   FLAG:<hex>
    Direction : SERVEUR → CLIENT
    Condition : identifiant ADMIN + RESP correct

    <hex> est le FLAG de session chiffré (voir section 6.2).

5.6 DENIED

    Syntaxe :   DENIED
    Direction : SERVEUR → CLIENT

    Émis dans les cas suivants :
      - Identifiant inconnu (phase AUTH)
      - Réponse incorrecte (phase RESP)
      - Délai de réponse dépassé (voir section 7)


6.  Fonctions cryptographiques

6.1 Calcul de RESP

    Pour chaque caractère SECRET[i], le byte correspondant est obtenu
    par une opération bit à bit XOR entre la valeur ASCII du caractère
    et la valeur du CHALLENGE.

    L'ensemble des bytes ainsi produits est encodé en hexadécimal,
    chaque byte sur exactement 2 caractères (padding zéro si nécessaire),
    puis concaténé sans séparateur pour former <hex>.

6.2 Chiffrement du FLAG

    Le FLAG est chiffré selon le même principe que RESP, à la différence
    que l'opération est appliquée entre chaque caractère du FLAG et le
    caractère correspondant du SECRET.

    Si le FLAG est plus long que le SECRET, le SECRET est répété
    cycliquement. L'encodage final est identique à celui de RESP.

6.3 Propriété d'inversibilité

    L'opération bit à bit utilisée dans ce protocole est symétrique :

        Si  A OP B = C
        Alors  C OP B = A

    Cette propriété s'applique aussi bien au calcul de RESP qu'au
    déchiffrement du FLAG.


7.  Contraintes temporelles

    Le SERVEUR ferme la session si aucune réponse RESP n'est reçue
    dans un délai de 5 secondes suivant l'émission du CHALLENGE.


8.  Exemples d'échanges complets

    Accès USER :
        CLIENT   →   AUTH:USER
        SERVEUR  →   CHALLENGE:<n>
        CLIENT   →   RESP:<hex>
        SERVEUR  →   OK

    Accès ADMIN :
        CLIENT   →   AUTH:ADMIN
        SERVEUR  →   CHALLENGE:<n>
        CLIENT   →   RESP:<hex>
        SERVEUR  →   FLAG:<hex>


9.  Considérations de sécurité

    Le mécanisme de CHALLENGE aléatoire empêche toute attaque par rejeu.
    Le SECRET ne transite jamais en clair sur le canal radio.
    L'utilisation d'une opération bit à bit seule ne constitue pas
    un chiffrement fort au sens cryptographique moderne.


10. Accès au système

    Système réel       :  Radio 2.4 GHz, canal 7
